### 1. EDD + Best-Fit Greedy Algorithm  

주어진 `baseline_greedy.py`의 구조를 살펴보겠다.  
우선 EDD는 Earliest Due Date의 줄임말로, 마감일이 빠른 순서대로 처리한다는 의미이다.  
Best-Fit Greedy는 그 순간 가장 적합한 것을 고르는 탐욕 알고리즘으로, 여러 후보가 있을 때 당장 가장 좋은 점수를 선택한다.  
또한 Post-Hoc Repair를 적용했다고 되어 있는데, 이는 나중에 수정한다는 의미로 위 방식대로 배치한 후 `check_feasibility`를 위반하는 블록들을 다시 배치한다.  

흐름은 2개의 Phase로 이루어진다.  
#### Phase-1: EDD 순서로 공격적 탐욕적 배치  
   block들은 가장 빠른 마감일 순서대로 정렬된다.  
   각 block에 대해 모든 **bay, orientation, position, time-slot** 조합에 점수를 매기고 가장 효율적인 조합을 확정한다.  
   2편에서 보았던 `check_entry`와 `check_exit`는 이 Phase 1이 진행되는 도중 해당 block의 조합을 결정할 때 수행된다.  
   그러나 주의할 점은, 모든 블록은 ENTRY/EXIT을 가져야 하는 제약으로 인해 탐색한 후보 조합 중 크레인 경로 검사를 통과하는 것이 하나도 없을 경우,
   Fallback 배치를 하게 된다.  
   &rarr; Fallback 배치란, 해당 block의 가장 선호도가 높은 bay 순서대로 검사해 비는 시간에 최소 유효 위치에 배치하는 것이다. 아래 코드에서 볼 수 있다.
   ```python
   for bay_id in sorted(range(n_bays), key=lambda j: prefs[j], reverse=True):  # 선호도가 높은 bay부터 순서대로 시도
        bay = bays[bay_id]
        for oi in range(len(blk_data["shape"])):  # 각 방향(orientation)을 순서대로 시도
            bb = _block_bbox(blk_data, oi)  # block의 바운딩박스를 계산 
            lx0, ly0, lx1, ly1 = bb
            px_lo = math.ceil(-lx0)  # 유효한 x 위치의 하한
            px_hi = math.floor(bay.width  - lx1)  # 유효한 x 위치의 상한
            py_lo = math.ceil(-ly0)  # 유효한 y 위치의 하한
            py_hi = math.floor(bay.height - ly1)  # 유효한 y 위치의 상한
            if px_lo > px_hi or py_lo > py_hi:
                continue  # no valid integer position for this orientation  # 이 방향으로는 정수 위치가 존재하지 않으므로 다음 방향 시도
            px = max(0, px_lo)  # bay 좌하단에 붙는 최소 유효 x 좌표 선택
            py = max(0, py_lo)  # bay 좌하단에 붙는 최소 유효 y 좌표 선택
            entry = _empty_bay_entry(bay_schedule[bay_id], r_time, proc)  # bay가 완전히 비는 가장 이른 시간대를 탐색
            return (bay_id, px, py, oi, entry, entry + proc)
   ```
   이 경우 항상 feasible한 이유
   1. bay가 비어있는 시간을 반환한다. 즉, entry_time, exit_time 모두 bay가 비어있어 장애물이 없다.  
      &rarr; 따라서, 만약 block이 bay에 들어가지 못하는 경우라면 어떤 방향으로도 해당 bay에 들어갈 수 없는 크기임을 알 수 있다.
   2. 최소 유효 위치는 block의 바운딩 박스가 bay의 왼쪽 아래 모서리에서 시작하도록 보장하므로, bay 경계를 넘지 않는다.

   만약 Fallback 배치로도 block이 어떤 bay에도 들어갈 수 없다면, 베이스라인 코드는 현재 RuntimeError로 멈추게 되어 있다.  


####  Phase-2: 반복적 repair  
Phase-1의 solution에 대해 `check_feasibility` 함수가 호출된다.  
&rarr; 조건을 위반하는 block이 있다면 EDD 순서대로 재배치 되며, 여기에는 2가지 모드가 있다.  

- greedy  
  greedy가 기본값이며, 이 경우 조건을 위반하는 block은 현재 solution에서 제거되고, 동일한 Phase-1을 거쳐 재배치된다.  
  이 과정에서 block이 2번째 repair에서 다시 나타나면, `_force_place`함수, 즉 위에서 말한 Fallback 배치를 하게 된다.  
  시간 제한의 90% 이후에 오는 블록도 `_force_place` 함수로 전송된다.
    ```python
  for bid in to_repair:
                repaired_counts[bid] = repaired_counts.get(bid, 0) + 1  # 이 블록이 repair 된 횟수 누적
                if repaired_counts[bid] > 1:
                    forced_ids.add(bid)  # 두 번째 이상 반복 위반되면 Fallback
  ```
  특정 block이 repair 된 횟수를 누적해, 2번째 이상일 시 Fallback 배치가 되는 것을 확인할 수 있다.

- simple  
  각 위반 block은 현재 (bay, orientation, position) 조합은 그대로 유지한 채, 그 bay 안에서 전체 처리 시간 동안 완전히 비어 있는 다음 시간대로만 이동한다.   
  bay의 경계를 넘어가는 경계 위반과, 두 block이 충돌하는 공간 충돌을 검사하는 `check_feasibility`의 Stage 4를 위반하는 경우 position을 (0,0)으로 재설정한다.

   

### 2. Challenges (막혔던 부분)  
1. Fallback 배치의 최소 유효 위치  
   Fallback 배치의 설명으로 주어진 최소 유효 위치에 대한 것이다.   
   먼저, Fallback의 경우 block은 선호도가 높은 bay 순서대로 왼쪽 아래 모서리에 들어갈 수 있는지 검사한다고 했는데, 이때 'block의 기준점(reference point)이 (0,0)에 속한다는 것인가?' 생각했다.  
   또한 그 경우 어째서 최소 유효 위치에 배치하면 무조건 feasible(bay의 경계를 넘지 않음)함을 보장할 수 있는지 의문이 들었다.  
   이는 위에서와 같이 코드를 보았을 때 기준점을 기준으로 왼쪽 아래 모서리에 들어가는 것이 아닌, 바운딩 박스를 기준으로 왼쪽 아래 모서리에 맞물리도록 하므로 기준점은 방향(orientation)에 따라 다른 위치에 존재할 수 있다는 것을 알게 되었다.  
   또한, 해당 bay가 비어있는 시간에 그러한 규칙으로 block을 배치하기 때문에 항상 feasible할 수 있다는 것도 납득할 수 있었다.  

2. Fallback 배치에서도 모든 bay에 대해 맞는 방향이 없는 경우 (degenerate instance) 처리 방법  
   현재 베이스라인 코드의 docstring에는 이 경우 선호 베이에서 orientation 0으로 배치된다고 작성되어 있다.  
   그러나 실제로 코드의 구조를 살펴본 결과, degenerate instance는 그렇게 강제로 배치되는 것이 아니라 RuntimeError가 발생하도록 되어있었다.  
   ```python
   raise RuntimeError(
        f"_force_place: block {bi} has no valid integer position in any bay "
        f"-- instance validation should have caught this."
    )
   ```
   주석을 읽는 것에 그치지 않고 코드 구조를 직접 확인함으로써 실제 베이스라인 알고리즘이 어떻게 돌아가는지 이해할 수 있었다.  
   또한, 어떤 block이 어떤 시간에도 어느 bay에 배치될 수 없는 경우는 올바른 문제 인스턴스에서는 발생해서는 안되는 경우이므로, 이 부분에 있어서도 주석의 설명보다는 현재 코드의 구조가 더 적절하다고 생각한다.  


3. 반복적 repair의 방식  
   Phase-2의 기본 greedy 모드에서, 조건을 위반하는 block이 제거된 후 재배치 된다고 하였다.
   처음에는 '이 위반한 block들은 위반하지 않는 block들이 exit한 후, 비어있는 베이에 새로 배치되는 것인가?' 생각했다.
   그러나 코드를 뜯어본 결과, 그렇지 않다는 것을 알 수 있었다.
   
   재배치는 아래와 같이 이루어진다.  
```python
            for bid in to_repair:
                assignments.pop(bid, None)  # 위반 block들을 현재 배치에서 제거

            n_bays = len(bays)
            bay_placed:    list[list[Block]]            = [[] for _ in range(n_bays)]
            bay_schedule2: list[list[tuple[int, int]]]  = [[] for _ in range(n_bays)]
            bay_loads:     list[float]                  = [0.0] * n_bays
```
   위반한 block들을 현재 배치에서 제거한다.  
   * `bay_placed`: 현재 배치된 block들이 담길 bay와 동일한 개수의 list이다.  
   * `bay_schedule2`: 현재 배치된 block들의 time-slot이 담길 bay와 동일한 개수의 list이다.  
   * `bay_loads`: bay별 누적 작업량을 담는다.  

```python
            for a in assignments.values():  # 상태를 재구성
                bid_a  = a["block_id"]
                bay_id = a["bay_id"]
                blk    = Block(block_id=bid_a, block_data=blocks_data[bid_a],
                               x=int(a["x"]), y=int(a["y"]), orient_idx=a["orient_idx"])
                bay_placed[bay_id].append(blk)
                bay_schedule2[bay_id].append((a["entry_time"], a["exit_time"]))
                bay_loads[bay_id] += blocks_data[bid_a]["workload"]
```
   그 후 위반하지 않은 block들의 position, time-slot을 유지한 채 상태를 재구성 한다.  
   다음으로 Phase-1과 동일한 탐색을 하는 `_place_blocks`로 넘어가기 때문에, 위반 block들을 빈 bay에서 처음부터 배치하는 것이 아니라 이미 자리 잡은 다른 block들의 사이에서 빈틈을 찾는 방식으로 작동한다는 것을 알 수 있었다.
