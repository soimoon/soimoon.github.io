이번에는 이어서 유틸리티 코드의 핵심 함수와 메서드들을 분석해보겠다.  

### 1. Utility Code Analysis (유틸리티 코드 분석)  

주어진 `utils.py`에는 4가지의 주요 함수가 있다.  

1. **`check_collisions(bay, blocks) -> list[CollisionResult]`**  
   &rarr; "현재" bay에 존재하는 모든 block 쌍을 조사해서 서로 공간이 겹치는지 검사한다.  
   &rarr; 속도를 높이기 위해 Shapely 라이브러리로 다각형을 정밀하게 대조하기 전에, AABB(상자)로 안 겹치는 block들을 빠르게 걸러낸다. (Early-exit)  
   &rarr; 블록 A의 k층과 블록 B의 k층만 1:1로 비교한다.  

2. **`check_entry(bay, blocks, new_block) -> list[EntryObstruction]`**  
   &rarr; 크레인 입고 가능 여부를 체크한다.  
   1. new_block이 주변 block과 부딪히지 않는지 1편에서 보았던 $$j \ge k$$ 규칙을 적용해 검사한다.  
   2. 또한, bay 밖으로 탈출하지 않았는지도 검사한다.  

3. **`check_exit(bay, blocks, target_block) -> list[EntryObstruction]`**  
   &rarr; 크레인 출고 가능 여부를 체크한다.  
   &rarr; check_entry와 대칭되는 로직을 쓴다. 단, target_block에 대해 검사하는 레이어는 동일하게 $$j \ge k$$ 규칙이 사용된다.  

4. **`check_feasibility(prob_info, solution) -> dict`**  
   &rarr; solution을 받아서 5단계에 걸쳐 제약조건을 전수 조사하는 함수이다.  
   &rarr; Feasible하면 최종 스코어(목적함수 값)를 반환한다.  


주요 함수 이외에도 다음과 같은 함수와 메서드가 있다.  

5. **Block 클래스의 메서드**  
   1. `bounding_rect()`  
      &rarr; 전체 블록의 AABB(축경계 바운딩박스)를 반환한다. 위의 check_collisions, check_entry, check_exit에서 Shapely로 정밀 대조하기 전에 빠르게 AABB 검사를 하는데 사용된다.  
   
   2. `layers_at_pos()`  
      &rarr; 주어진 문제 인스턴스 JSON에서, block들의 각 방향 별 레이어 0의 첫번째 꼭짓점은 [0,0]이다. 이를 기준점이라 한다.  
      &rarr; 어떤 block을 특정 방향으로 (x,y)에 놓아 Block 인스턴스를 생성하면, `__post_init__`이 실행되어 모든 레이어를 (x,y)만큼 이동한 결과를 저장한다.  
      &rarr; `layers_at_pos`는 그 값을 즉시 반환한다. 이 값은 check 함수들이 겹치는 부분을 탐색하는 데 사용되며, AABB를 얻는데도 사용된다.  
      
   3. `orientations()`, `num_orientations()`, `current_orient()`, `orientation_index()`  
      &rarr; 차례로 block이 가질 수 있는 모든 방향을 반환, 방향의 개수 반환, 현재 선택된 방향의 shape 항목을 반환, 문제 JSON의 방향 식별값을 반환하는 역할을 한다.  
      
   4. `from_instance(cls, block_id, instance, x=0, y=0, orient_idx=0) -> "Block"`  
      &rarr; 인스턴스 딕셔너리에서 block 원본 데이터를 찾아 Block 객체를 생성한다.  

6. **`Bay.contains_block()`**  
   &rarr; block의 전체 영역이 bay 내에 포함될 경우 True를 반환한다.  
   &rarr; block의 AABB를 기준으로 `0 <= min_x`, `max_x <= width`, `0 <= min_y`, `max_y <= height`를 확인한다.  

7. **`CollisionResult` / `EntryObstruction`의 Field**  

   단순히 충돌 여부가 아닌, 어디서 막힌 것인지 확인하는데 활용할 수 있다.  

   1. `.area`  
      &rarr; 겹친 면적으로, 겹친 정도를 확인할 수 있다.  
   2. `.is_sweep`  
      &rarr; 최종 위치 충돌(False)인지 경로상 스윕 충돌(True)인지 반환하므로, repair 로직에서 위치와 시간 중 어떤 것을 조절해야하는지 판단할 수 있다.  
   3. `.new_layer` / `.exist_layer`  
      &rarr; 어느 층에서 막혔는지 알 수 있다.

### 2. Challenges (막혔던 부분)  
**1. `orientation_index()`의 역할**  

Block 생성자에서 `orient_idx`도 블록의 방향을 의미하게 되는데, 이것과 `orientation_index()`이 반환하는 값에 어떤 차이가 있는지 궁금증이 생겼다.  
이는 문제 JSON의 "shape" 구조를 다시 보며 알 수 있었다. `orient_idx`는 리스트에서 몇번째 원소를 가져올지 가리키는 인덱스 값이다.  

만약 이 값을 그대로 사용한다면 인덱스 순서에 의존하게 되어, 문제 JSON의 `orientation`이 1, 2, 3 순서가 아닌 1, 3, 5 처럼 띄엄띄엄 생성되는 경우 의도한 방향을 가져오지 못할 수 있다.  
따라서 `orientation_index()`는 정확한 방향 식별값을 반환하는 역할을 한다는 것을 알 수 있었다.  

**2. `contains_block()`이 검사하는 범위**  
`contains_block()`는 block의 AABB를 기준으로 해당 block이 bay 안에 포함되는지 확인한다고 했다.  
따라서 처음에는 블록의 정확한 레이어 별 다각형 모양을 확인하는 것이 아니기 때문에, 'check 함수들이 Shapely를 사용하기 전 일차적으로 검사하던 것처럼 일차적으로 걸러내는 역할인가?' 라는 생각을 했다.  
bay가 직사각형이기 때문에, AABB가 bay 안에 들어간다면 당연히 block도 bay안에 들어갈 것이니 이 부분은 문제가 없다.  
그러나 'AABB는 bay에서 조금 나갔지만, 막상 정밀하게 검사하면 block은 나가지 않은 경우도 있지 않은가?' 라는 의문이 든 것이다.  

이 부분은 아래의 `_bounding_box`가 AABB를 계산하는 방법을 다시 보며 의문이 해소되었다.  

<br>

```python
def _bounding_box(verts: list) -> tuple[float, float, float, float]:
    xs = [v[0] for v in verts]  # 모든 꼭짓점의 x 좌표 목록
    ys = [v[1] for v in verts]  # 모든 꼭짓점의 y 좌표 목록
    return min(xs), min(ys), max(xs), max(ys)  # (최소x, 최소y, 최대x, 최대y) 형태의 바운딩박스 반환
```

<br>

처음에 무심코 상자 모양의 AABB와 실제 block에 차이가 있을 것이라고 생각한 것과 달리,  
여기서 최대 x값과 최대 y값을 보면 여유 없이 실제 다각형 꼭짓점들의 좌표에서 가장 큰 값이 반환된다는 것을 알 수 있다.  
따라서 AABB가 bay를 넘어갔다는 것은, 실제 다각형의 꼭짓점이 넘어갔다는 뜻이 되므로 AABB가 나가는 경우에도 무조건 block이 나갔다고 간주할 수 있다는 것을 알게 되었다.
